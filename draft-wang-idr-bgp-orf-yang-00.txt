      YANG Data Model for BGP Outbound Route Filtering
                  draft-wang-idr-bgp-orf-yang-00

Abstract

   This document defines YANG data models for BGP Outbound Route
   Filtering (ORF). The first module, ietf-bgp-orf, provides a
   generic ORF anchor under the BGP neighbor AFI/SAFI hierarchy.
   The second module, ietf-bgp-vpn-prefix-orf, augments the generic
   ORF anchor for VPN Prefix ORF and provides per-VRF quota and
   overload control.

1. Introduction

   RFC 5291 defines a BGP-based mechanism that allows a BGP speaker
   to send Outbound Route Filters (ORFs) to a peer so that the peer
   can constrain outbound route advertisements to that speaker. The
   ORF capability is negotiated per AFI/SAFI and per ORF Type.

   Existing ORF types include Address Prefix ORF, CP-ORF, and VPN
   Prefix ORF. VPN Prefix ORF introduces VPN-specific semantics
   including Route Distinguisher, Route Target, Source PE, per-VRF
   quota, and overload handling.

   This document defines two YANG modules:

      * ietf-bgp-orf

      * ietf-bgp-vpn-prefix-orf

   The generic ORF module intentionally models only those semantics
   that are common to ORF mechanisms, including ORF Type, send/receive
   direction, entry limits, refresh behavior, and operational state.
   VPN Prefix ORF-specific semantics are modeled by a separate
   augmentation module.

   This split avoids adding VPN-specific ORF semantics to the base BGP
   YANG model while keeping ORF configuration anchored to the BGP
   protocol context in which ORF capability is negotiated and applied.

2. Terminology

   ORF:
      Outbound Route Filtering.

   ORF Type:
      The ORF Type value maintained in the IANA BGP Outbound Route
      Filtering (ORF) Types registry.

   VPN Prefix ORF:
      The VPN Prefix Outbound Route Filter defined by
      I-D.ietf-idr-vpn-prefix-orf.

   Source PE:
      The PE from which VPN routes are originated.

   Quota:
      The maximum number of VPN routes permitted for a configured
      scope, such as a VRF, RD, Source PE, or <RD, Source PE> pair.

   Overload:
      A condition in which the number of VPN routes exceeds a
      configured quota or threshold.

3. Design Objectives

   The model follows these design objectives:

   * Generic ORF configuration is anchored under BGP neighbor
     AFI/SAFI, because ORF capability is negotiated in that protocol
     context.

   * ORF type-specific information is modeled through augmentation
     rather than being embedded in the base BGP model.

   * VPN Prefix ORF quota and overload policy are modeled under
     network-instance/VRF because overload is normally evaluated at
     the receiving VRF.

   * VPN Prefix ORF entry state is attached under generic ORF Type 66.

   * The generic ORF module may be reused by future ORF types without
     requiring VPN Prefix ORF-specific nodes to be added to the base
     BGP YANG model.

4. YANG Tree Structure

4.1. Generic ORF Anchor

   The generic ORF anchor is attached under BGP neighbor AFI/SAFI.

   +--rw orf
      +--rw enabled? boolean
      +--rw orf-type* [type]
         +--rw type uint8
         +--rw type-identity? identityref
         +--rw enabled? boolean
         +--rw direction? enumeration
         +--rw max-received-entries? uint32
         +--rw max-originated-entries? uint32
         +--rw default-when-to-refresh? enumeration
         +--ro state
            +--ro negotiated? boolean
            +--ro negotiated-direction? enumeration
            +--ro received-entry-count? uint32
            +--ro originated-entry-count? uint32
            +--ro last-received-time? yang:date-and-time
            +--ro last-originated-time? yang:date-and-time
            +--ro last-error? string

4.2. VPN Prefix ORF Augmentation

   VPN Prefix ORF augments generic ORF Type 66.

   +--rw orf
      +--rw orf-type* [type]
         +--rw type = 66
         +--ro vpn-prefix-orf
            +--ro originated-vpn-prefix-orf-entry* [entry-id]
            +--ro received-vpn-prefix-orf-entry* [entry-id]

4.3. Per-VRF VPN Prefix ORF Policy

   VPN Prefix ORF quota and overload policy are attached to the VRF or
   network instance.

   +--rw network-instances
      +--rw network-instance* [name]
         +--rw vpn-prefix-orf
            +--rw enabled? boolean
            +--rw mode? enumeration
            +--rw local-vrf-prefix-limit? uint32
            +--rw default-source-pe-quota? uint32
            +--rw source-pe-quota* [rd source-pe-id]
            +--rw overload-threshold-percent? uint8
            +--rw clear-threshold-percent? uint8
            +--rw auto-generate-orf? boolean
            +--rw overload-process-method? enumeration
            +--ro state
               +--ro overload? boolean
               +--ro current-prefix-count? uint32
               +--ro active-quota? uint32
               +--ro last-overload-time? yang:date-and-time

  yang-version 1.1;

  namespace "urn:ietf:params:xml:ns:yang:ietf-bgp-orf";

  prefix bgp-orf;

  import ietf-bgp {

      "I-D.ietf-idr-bgp-model: YANG Model for Border Gateway
       Protocol (BGP-4).";

    prefix yang;
    reference
      "RFC 6991: Common YANG Data Types.";

    "IETF IDR Working Group";

     WG List:  <mailto:idr@ietf.org>";

  description
    "This module defines a generic YANG anchor for BGP Outbound
     Route Filtering (ORF). Type-specific ORF modules may augment
     this module.

     Copyright (c) 2026 IETF Trust and the persons identified as

  }

    description
      "Support for BGP Outbound Route Filtering.";
  }

  typedef orf-type {
    type uint8 {
      range "0..255";
    }
    description
      "BGP ORF Type value as maintained by IANA.";
    reference
      "IANA BGP Outbound Route Filtering (ORF) Types registry.";
  }

  typedef orf-send-receive {
    type enumeration {
      enum receive {
        value 1;
        description
          "The local speaker is willing to receive ORF entries from
           its peer.";
      }
      enum send {
        value 2;
        description
          "The local speaker would like to send ORF entries to its
           peer.";
      }
      enum both {
        value 3;
        description
          "The local speaker supports both sending and receiving ORF
           entries.";
      }
    }
    description
      "ORF Send/Receive field.";
    reference
      "RFC 5291.";
  }

  typedef orf-action {
    type enumeration {
      enum add {
        value 0;
        description
          "Add an ORF entry.";
      }
      enum remove {
        value 1;
        description
          "Remove an ORF entry.";
      }
      enum remove-all {
        value 2;
        description
          "Remove all ORF entries for this ORF type.";
      }
    }
    description
      "ORF Action field.";
    reference
      "RFC 5291.";
  }

  typedef orf-match {
    type enumeration {
      enum permit {
        value 0;
        description
          "Permit matching routes.";
      }
      enum deny {
        value 1;
        description
          "Deny matching routes.";
      }
    }
    description
      "ORF Match field.";
    reference

  }

        value 1;

      }
    }
    description
      "When-to-refresh behavior for ORF.";

          description
            "Operational state for this ORF type.";

            type boolean;
            description
              "Indicates whether this ORF type has been successfully

          }

          leaf negotiated-direction {
            type orf-send-receive;
            description
              "Negotiated ORF direction.";
          }

          leaf received-entry-count {
            type uint32;
            description
              "Number of active ORF entries received from the peer.";
          }

          leaf originated-entry-count {
            type uint32;
            description
              "Number of active ORF entries originated toward the
               peer.";
          }

          leaf last-received-time {
            type yang:date-and-time;
            description
              "Last time an ORF entry of this type was received.";
          }

          leaf last-originated-time {
            type yang:date-and-time;
            description

          leaf last-error {
            type string;
            description
              "Last ORF-related error for this ORF type.";
          }
        }
      }
    }
  }

   * The exact augment target path may need to be aligned with the

        + "bgp:afi-safis/bgp:afi-safi" {
    description
      "Attach generic ORF controls to BGP neighbor AFI/SAFI.";

    uses orf-anchor;
  }

7. Security Considerations

   The YANG modules defined in this document are designed to be
   accessed through network management protocols such as NETCONF and
   RESTCONF. The lowest NETCONF layer is the secure transport layer,
   and the mandatory-to-implement secure transport is SSH. The lowest
   RESTCONF layer is HTTPS, and the mandatory-to-implement secure
   transport is TLS.

   The Network Configuration Access Control Model (NACM) provides the
   means to restrict access for particular NETCONF or RESTCONF users
   to a preconfigured subset of all available NETCONF or RESTCONF
   protocol operations and content.

   There are a number of data nodes defined in these YANG modules that
   are writable, creatable, or deletable. These data nodes may be
   considered sensitive or vulnerable in some network environments.
   Write operations to these data nodes without proper protection can
   have a negative effect on network operations.

   In particular, incorrect configuration of ORF direction, entry
   limits, or VPN Prefix ORF quota may cause routes to be
   unintentionally filtered, withdrawn, or suppressed. Such
   misconfiguration may result in loss of VPN reachability.

   The following writable data nodes are security sensitive:

      /orf/enabled

      /orf/orf-type/enabled

      /orf/orf-type/direction

      /orf/orf-type/max-received-entries

      /orf/orf-type/max-originated-entries

      /network-instance/vpn-prefix-orf/local-vrf-prefix-limit

      /network-instance/vpn-prefix-orf/default-source-pe-quota

      /network-instance/vpn-prefix-orf/source-pe-quota

      /network-instance/vpn-prefix-orf/overload-threshold-percent

      /network-instance/vpn-prefix-orf/clear-threshold-percent

      /network-instance/vpn-prefix-orf/auto-generate-orf

      /network-instance/vpn-prefix-orf/overload-process-method

   Unauthorized modification of these nodes can cause excessive route
   suppression or route advertisement, route churn, VPN reachability
   loss, or overload oscillation.

   Some readable data nodes may also be sensitive. Read access to
   operational state may disclose BGP peering structure, VPN route
   scale, Source PE identifiers, Route Distinguisher values, Route
   Target values, overload conditions, and route filtering behavior.

   Operators should restrict read and write access to authorized
   management clients and should apply suitable logging and audit
   controls to all configuration changes that affect ORF behavior.
